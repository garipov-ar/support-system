- 1.  Требования к разрабатываемой системе
        1.  Требования к структуре системы в целом

1.4.1.1 Перечень подсистем, их назначение и основные характеристики

1.4.1.1.1 Подсистема управления контентом (CMS). Назначение: централизованное хранение, структурирование и актуализация технической документации. Основные характеристики:

1.  поддержка иерархической структуры категорий (поддерживаемые типы: инструкции, прошивки, схемы, прайс-листы);
2.  поддержка версионности: хранение истории файлов с возможностью доступа к любой версии;
3.  визуальная индикация актуальности контента;

1.4.1.1.2 Подсистема интеграции с Telegram-ботом. Назначение: обеспечение мобильного доступа к информации в полевых условиях. Основные характеристики:

1.  интерфейс на базе интерактивных меню и команд;
2.  навигация по дереву категорий, полностью синхронизированная с CMS;
3.  полнотекстовый поиск по заголовкам и описаниям документов;
4.  система подписок на уведомления об обновлениях в выбранных разделах;

1.4.1.1.3 Подсистема администрирования (админ-панель). Назначение: управление доступом, учетными записями и аудит действий. Основные характеристики:

1.  гибкая ролевая модель (3 роли: специалист, контент-менеджер, администратор);
2.  ведение автоматизированного журнала аудита (Audit Log) всех значимых действий;

1.4.1.1.4 Подсистема аналитики и мониторинга. Назначение: сбор и визуализация статистики использования системы. Основные характеристики:

1.  дашборд с ключевыми метриками: активность пользователей, топ поисковых запросов, среднее время ответа;
2.  мониторинг состояния интеграции с Telegram API (Watchdog);
3.  логирование времени отклика для контроля производительности;

1.4.1.2 Требования к способам и средствам обеспечения информационного взаимодействия компонентов системы

1.4.1.2.1 Система должна функционировать в рамках единого информационного пространства с централизованной реляционной базой данных, обеспечивающей согласованность и целостность информации между всеми подсистемами.

1.4.1.2.2 Система должна обеспечивать межкомпонентное взаимодействие посредством RESTful API с использованием формата обмена данными JSON и протокола передачи данных HTTPS.

1.4.1.2.3 Система должна шифровать все запросы между клиентской и серверной частями с использованием протоколов SSL/TLS.

- - 1.  Требования к функциям, выполняемым системой

1.4.2.1 Функции, подлежащие автоматизации

1.4.2.1.1 Подсистема управления контентом (CMS) должна обеспечивать следующие функции:

1.  добавление новых файлов различных форматов (инструкций, прошивок, конфигураций, прайс-листов) через веб-интерфейс;
2.  управление деревом категорий через MPTT (префиксные деревья) для быстрой навигации;
3.  назначение документов конкретным единицам оборудования;
4.  управление версиями: каждая загрузка файла создает новую запись в истории версий;

1.4.2.1.2 Подсистема интеграции с Telegram-ботом должна обеспечивать следующие функции:

1.  автоматическая регистрация пользователя при первом запуске (сбор имени и email);
2.  обязательное подтверждение согласия на обработку персональных данных (ФЗ-152);
3.  поиск документов по частичному совпадению текста;
4.  автоматическая рассылка уведомлений подписанным пользователям при выходе новой версии документа;

1.4.2.1.3 Подсистема администрирования должна обеспечивать следующие функции:

1.  управление учетными записями: создание, блокировка и редактирование профилей сотрудников;
2.  разграничение прав доступа на основе групп: назначение ролей ("Администратор", "Контент-менеджер") с разными уровнями разрешений на чтение/запись данных;
3.  журналирование действий: автоматическая фиксация каждого факта входа в систему, изменения документов, удаления версий или изменения настроек;
4.  модерация пользовательских данных: возможность просмотра и удаления информации о пользователях бота по требованию;
5.  фильтрация системных журналов: поиск записей аудита по дате, типу события, IP-адресу или конкретному пользователю.

1.4.2.1.4 Подсистема аналитики и мониторинга должна обеспечивать следующие функции:

1.  автоматический сбор статистики: учет каждого запроса к боту, времени отклика сервера и количества возвращенных результатов поиска;
2.  формирование дашбордов: визуализация метрик (активность за 7/30 дней, среднее время отклика, объем занимаемого дискового пространства) в виде графиков;
3.  анализ эффективности поиска: выявление топ-10 популярных поисковых запросов и фиксация запросов, не давших результатов (для оптимизации базы знаний);
4.  контроль состояния (Health Check): визуальная индикация статуса Telegram API, БД и фоновых задач Celery в режиме реального времени;
5.  формирование отчетов по активности пользователей: выгрузка данных в форматах Excel/PDF для последующего анализа руководством

1.4.2.2 Временной регламент реализации функций

1.4.2.2.1 Система должна обеспечивать время отклика на запрос через Telegram-бот не более 2 секунд при стабильном интернет-соединении.

1.4.2.2.2 Система должна обеспечивать формирование дерева разделов в боте не более чем за 1 секунду.

1.4.2.2.3 Система должна обеспечивать рассылку уведомлений об обновлении контента в течение 1 минуты с момента публикации.

1.4.2.3 Требования одновременности выполнения группы функций

1.4.2.3.1 Система должна выдерживать до 100 одновременных пользователей без снижения производительности.

1.4.2.4 Критерии отказов

1.4.2.4.1 Отказом системы считается нарушение выполнения любой из функций, указанных в п. 1.4.2.1, при условии:

1.  корректной работы технических средств (пользовательское устройство, интернет-канал, сервер);
2.  соблюдения пользователем правил эксплуатации (корректный ввод команд, успешная авторизация).

1.4.2.4.2 Отказом также считается:

1.  возврат устаревшей версии документа без соответствующего уведомления;
2.  невозможность доступа к документу, помеченному в CMS как «актуальный»;
3.  потеря или искажение метаданных (версия, оборудование) при загрузке или обновлении файла.

1.4.3.1 Требования к математическому обеспечению

Требования к математическому обеспечению не предъявляются.

1.4.3.2 Требования к информационному обеспечению

1.4.3.2.1 Система должна использовать в качестве системы управления базами данных PostgreSQL версии 14.0 или выше..

1.4.3.2.2 Система должна обеспечивать хранение следующих типов данных:

1.  техническая документация в электронном виде (инструкции, прошивки, конфигурации, прайс-листы);
2.  метаданные документов (категория, оборудование, версия, дата обновления, автор);
3.  журнал операций (логи запросов, загрузки файлов, входы в систему, изменения контента);
4.  данные пользователей (Telegram ID, ФИО, email, роль, дата регистрации).

1.4.3.2.3 Система должна обеспечивать выполнение резервного копирования данных в следующем режиме:

1.  ежедневное инкрементальное резервное копирование;
2.  еженедельное полное резервное копирование базы данных и медиафайлов;
3.  хранение резервных копий сроком не менее 3 месяцев.

1.4.3.2.4 Система должна обеспечивать хранение истории версий документов не менее 3 месяцев для возможности отката к предыдущим версиям.

1.4.3.3 Требования к лингвистическому обеспечению

1.4.3.3.1 Система должна использовать в качестве языков программирования Python (с фреймворком Django);

**1.4.3.4 Требования к программному обеспечению (ПО)**

1.4.3.4.1 Требования к серверному программному обеспечению.

1.4.3.4.1.1 Система должна функционировать в среде контейнеризации Docker под управлением ОС семейства Linux (рекомендуется Ubuntu Server 22.04 LTS).

1.4.3.4.1.2 Система должна использовать веб-сервер Nginx для обработки внешних запросов и терминирования SSL-трафика.

1.4.3.4.1.3 Система должна использовать брокер сообщений Redis для управления состояниями чат-бота и очередями задач.

1.4.3.4.2 Требования к клиентскому программному обеспечению.

1.4.3.4.2.1 Система должна быть доступна через официальное приложение Telegram на платформах Android, iOS, Windows и macOS.

1.4.3.4.2.2 Веб-интерфейс CMS должен корректно отображаться и функционировать в браузерах Google Chrome, Mozilla Firefox, Microsoft Edge и Safari последних версий.

1.4.3.4.3 Требования к обеспечению качества программного обеспечения.

1.4.3.4.3.1 Точность и полнота функциональности: ПО должно полностью соответствовать заявленным функциональным требованиям. Все функции должны работать корректно и без сбоев.

1.4.3.4.3.2 Система должна сопровождаться исходным кодом, соответствующим стандартам PEP8, с наличием комментариев к ключевым модулям и функциям.

1.4.3.4.3.3 Система должна быть сопровождаемой за счёт:

1.  высокого качества эксплуатационной документации;
2.  использования осмысленных имён переменных и функций;
3.  наличия комментариев в исходном коде;
4.  соблюдения единого стиля форматирования кода.

1.4.3.5 Требования к техническому обеспечению

1.4.3.5.1 Система должна функционировать на сервере со следующими минимальными характеристиками (расчёт на 100 одновременно работающих пользователей):

1.  процессор – не менее 2 ядер, 2.5 ГГц;
2.  оперативная память – не менее 4 ГБ;
3.  накопитель – SSD не менее 20 ГБ;
4.  сетевой адаптер – 1 Гбит/с.

1.4.3.5.2 Система должна быть доступна с клиентских устройств, соответствующих следующим требованиям:

1.  наличие современного веб-браузера или приложения Telegram;
2.  поддержка JavaScript и CSS3;
3.  разрешение экрана не менее 360×800 пикселей.

1.4.3.6 Требования к метрологическому обеспечению

Требования к метрологическому обеспечению не предъявляются.

1.4.3.7 Требования к организационному обеспечению

1.4.3.7.1 Эксплуатацию системы должны обеспечивать:

1.  выездные специалисты – конечные пользователи с базовыми цифровыми навыками, осуществляющие доступ к информации через мобильный интерфейс бота;
2.  служба поддержки – администраторы/контент-менеджеры с полными правами на структурирование и управление актуальностью контента;
3.  ИТ-отдел – технические специалисты, отвечающие за настройку серверного оборудования, выполнение резервного копирования и мониторинг работоспособности.

1.4.3.7.2 Система должна ограничивать доступ к серверным компонентам и панели управления исключительно для авторизованных лиц из состава персонала.

1.4.3.7.3 Перед началом промышленной эксплуатации системы администраторы и контент-менеджеры должны пройти инструктаж по работе с интерфейсами управления.

1.4.3.8 Требования к методическому обеспечению

1.4.3.8.1 Разработчик должен предоставить в составе системы следующую эксплуатационную документацию на русском языке;

1.  руководство пользователя (для выездных специалистов;
2.  руководство администратора (для службы поддержки и ИТ-отдела);
    - 1.  Общие технические требования к системе

1.4.4.1 Требования к численности и квалификации персонала и пользователей системы

1.4.4.1.1 Система должна быть интуитивно понятной и не требовать от конечных пользователей (специалистов) прохождения специального обучения, кроме ознакомления с краткой инструкцией.

1.4.4.1.2 Система должна быть интуитивно понятной и не требовать специальной подготовки для клиентов.

1.4.4.2 Требования к показателям назначения

1.4.4.2.1 Система должна сохранять целевое назначение на протяжении всего срока эксплуатации.

1.4.4.2.2 Срок эксплуатации системы не ограничен при условии своевременного обновления ПО и резервного копирования.

1.4.4.2.3 Система должна допускать настройку категорий, метаданных, прав доступа и уведомлений без участия разработчика.

1.4.4.3 Требования к надежности

1.4.4.3.1 Система должна обеспечивать целостность данных при сбоях за счет использования механизмов транзакций в СУБД.

1.4.4.3.2 Система должна выдерживать нагрузку до 100 одновременно работающих пользователей без снижения производительности.

1.4.4.3.3 Система должна обеспечивать целевой уровень доступности (Uptime) не менее 99.5%.

1.4.4.3.4 Система должна предотвращать дублирование документов за счёт блокировок на уровне базы данных.

1.4.4.4 Требования по безопасности

1.4.4.4.1 Система должна соответствовать требованиям Федерального закона №152-ФЗ «О персональных данных», включая:

1.  получение согласия на обработку данных при первом запуске бота;
2.  хранение идентификационных данных в зашифрованном виде;
3.  возможность удаления данных по запросу пользователя.

1.4.4.4.2 Система использовать протокол HTTPS для всех видов сетевого взаимодействия.

1.4.4.4.3 Система должна обеспечивать безопасное хранение паролей администраторов с использованием хэш-функции PBKDF2 со значением соли (стандарт Django).

1.4.4.5 Требования к эргономике и технической эстетике

1.4.4.5.1 Веб-интерфейс системы должен иметь адаптивную верстку для корректного отображения на экранах с шириной от 360 пикселей.

1.4.4.5.2 Интерфейс Telegram-бота должен обеспечивать доступ к документам не более чем за 3-4 клика по меню.

1.4.4.6 Требования к эксплуатации, техническому обслуживанию, ремонту и хранению компонентов системы

1.4.4.6.1 Эксплуатация системы должна осуществляться подготовленным персоналом (выездными специалистами и сотрудниками службы поддержки), прошедшим ознакомление с эксплуатационной документацией.

1.4.4.6.2 Техническое обслуживание системы (обновление программных модулей, резервное копирование, мониторинг) должно осуществляться ИТ-отделом организации или разработчиком.

1.4.4.6.3 Восстановление работоспособности системы после критических сбоев должно выполняться квалифицированными специалистами в рамках гарантийных обязательств или сервисного контракта.

1.4.4.6.4 Хранение программного кода и архивных копий данных должно осуществляться в защищенном хранилище с использованием многофакторной аутентификации.

1.4.4.6.5 Авторские права на исходный код, дизайн и архитектуру системы принадлежат разработчику и защищены действующим законодательством РФ.

1.4.4.6.6 Пользователь не имеет права передавать программное обеспечение третьим лицам, использовать его в качестве образца для создания аналогичных систем или модифицировать без согласования с разработчиком.

1.4.4.7 Требования к защите информации от несанкционированного доступа

1.4.4.7.1 Доступ к административным интерфейсам и базе данных должен осуществляться исключительно после успешной аутентификации по уникальному логину и паролю.

1.4.4.7.2 Пароли пользователей должны храниться в базе данных исключительно в виде криптографических хэшей (с использованием алгоритмов PBKDF2 или Argon2).

1.4.4.7.3 Все интерактивные формы и API-сервисы должны функционировать исключительно по протоколу HTTPS с использованием действующих SSL/TLS-сертификатов.

1.4.4.7.4 Доступ к серверному оборудованию на сетевом уровне должен быть ограничен с использованием межсетевых экранов и разрешен только для доверенных IP-адресов или через VPN-туннели.

1.4.4.7.5 Система должна обеспечивать автоматизированное ведение журналов событий безопасности (входы, изменения прав, попытки несанкционированного доступа).

1.4.4.8 Требования по сохранности информации при авариях

1.4.4.8.1 Система должна обеспечивать сохранность и целостность данных при программно-аппаратных сбоях, кратковременном отключении питания или сетевых ошибках.

1.4.4.8.2 В случае выхода из строя основного вычислительного узла данные должны быть восстанавливаемы из актуальных резервных копий.

1.4.4.8.3 Все операции по модификации данных должны выполняться в рамках транзакций СУБД для гарантированного соблюдения принципов ACID.

1.4.4.8.4 Резервное копирование должно выполняться:

1.  ежедневно (инкрементальное);
2.  еженедельно (полное);
3.  срок хранения копий - не менее 3 месяцев.

1.4.4.9 Требования к защите от влияния внешних воздействий

2.1.2.1 Анализ временного регламента реализации функций

Временной регламент реализации функций определяет требования к скорости выполнения основных операций системы и оказывает существенное влияние на удобство и эффективность работы пользователей. Для системы информационной поддержки специалистов службы технической поддержки соблюдение временных характеристик является критически важным, так как задержки при получении информации могут негативно сказаться на качестве и скорости выполнения служебных задач.

Среднее время отклика системы

Требование: среднее время отклика системы на пользовательские действия не должно превышать 2 секунд при штатной нагрузке.

Под временем отклика понимается интервал времени от момента получения запроса системой до формирования и отправки пользователю полного ответа. Данное требование распространяется на следующие операции:

- поиск и просмотр технической документации;
- получение инструкций и справочной информации через веб-интерфейс;
- получение ответов и навигацию по данным с использованием Telegram-бота;
- выполнение операций управления контентом через административный интерфейс.

Решения:

а) оптимизация запросов к базе данных PostgreSQL за счёт:

- использования индексов по полям, участвующим в поиске и фильтрации;
- минимизации количества сложных объединений таблиц;
- оптимизации структуры данных и запросов;

б) применение кэширования часто запрашиваемых данных с использованием Redis, что позволяет существенно сократить количество обращений к основной базе данных;

в) использование асинхронной обработки ресурсоёмких операций с применением фоновых задач Celery, что исключает блокировку пользовательских запросов;

г) сокращение объёма передаваемых данных за счёт передачи только необходимой информации в формате JSON;

д) при необходимости проведение нагрузочного тестирования системы для подтверждения соответствия требованиям по времени отклика.

Время выполнения длительных операций

Требование: операции, связанные с обработкой большого объёма данных (например, загрузка файлов, массовое обновление информации, формирование отчётов), не должны блокировать работу пользовательского интерфейса.

Решения:

а) вынести длительные операции в фоновую обработку с использованием очередей задач Celery;

б) реализовать механизм уведомления пользователя о завершении операции (например, через интерфейс системы или Telegram-бот);

в) обеспечить устойчивость фоновых задач к временным сбоям и возможность их повторного выполнения.

Время запуска и перезапуска системы

Требование: полное время запуска или перезапуска системы не должно превышать 5 минут.

Время запуска включает:

- инициализацию серверной части системы;
- установление соединений с базой данных и кэшем;
- запуск фоновых задач;
- восстановление рабочих состояний Telegram-бота.

Решения:

а) использование контейнерной архитектуры (Docker), позволяющей ускорить развертывание и перезапуск компонентов системы;

б) параллельная инициализация независимых сервисов (сервер приложения, Redis, Celery);

в) минимизация объёма данных, загружаемых при старте системы, за счёт предварительного кэширования и отложенной загрузки некритичных данных;

г) ведение логирования этапов запуска системы для выявления узких мест и последующей оптимизации.

К возможным рискам нарушения временного регламента относятся:

- рост времени отклика при увеличении числа пользователей;
- выполнение ресурсоёмких операций в основном потоке обработки запросов;
- деградация производительности базы данных при увеличении объёма данных.

Для минимизации указанных рисков необходимо:

а) регулярно анализировать показатели производительности системы;

б) масштабировать компоненты системы при увеличении нагрузки;

в) оптимизировать структуру базы данных и конфигурации сервисов;

г) использовать кэширование и асинхронную обработку операций.

2.1.2.2 Анализ требований одновременности выполнения группы функций

Требования к одновременности выполнения функций определяют способность системы обеспечивать корректную и устойчивую работу при одновременном выполнении операций несколькими пользователями и подсистемами. Для системы информационной поддержки специалистов службы технической поддержки данное требование является критически важным, так как система предполагает параллельную работу выездных специалистов, операторов службы поддержки и административного персонала.

Отсутствие принципиальных ограничений на количество пользователей

Требование: система не должна иметь принципиальных ограничений на количество одновременно работающих пользователей. Ограничения допускаются исключительно на уровне аппаратных ресурсов серверной инфраструктуры и сетевого обеспечения.

Данное требование распространяется на следующие виды одновременной работы:

- одновременный поиск и просмотр информации несколькими пользователями;
- параллельное использование веб-интерфейса и Telegram-бота;
- одновременное выполнение операций управления контентом и получения информации.

Решения:

а) использовать серверную архитектуру, поддерживающую параллельную обработку запросов;

б) обеспечить масштабируемость серверной части за счёт контейнеризации компонентов системы;

в) применять балансировку нагрузки между компонентами системы при увеличении количества запросов;

г) минимизировать блокирующие операции в серверной логике.

Обработка конкурирующих операций

Требование: при одновременном выполнении операций несколькими пользователями система должна обеспечивать целостность и согласованность данных. Конкурирующие операции не должны приводить к потере или искажению информации.

Решения:

а) использование транзакционных механизмов базы данных PostgreSQL для обеспечения атомарности операций;

б) реализация механизмов блокировки данных на уровне базы данных при выполнении операций изменения информации;

в) предотвращение конфликтов при одновременном редактировании данных за счёт проверки актуальности данных перед их сохранением;

г) использование очередей задач для последовательной обработки операций, требующих строгой последовательности выполнения.

Одновременное выполнение функций различными подсистемами

Требование: система должна обеспечивать корректную работу всех подсистем при их одновременном функционировании, включая веб-интерфейс, Telegram-бот и фоновые задачи.

Решения:

а) разделить обработку пользовательских запросов и выполнение фоновых операций с использованием Celery;

б) обеспечить независимость работы подсистем, чтобы сбои в одной из них не влияли на функционирование остальных;

в) использовать Redis для обмена состояниями и снижения нагрузки при одновременной работе компонентов.

Масштабируемость при росте нагрузки

Требование: система должна поддерживать возможность масштабирования при увеличении количества пользователей и объёма выполняемых операций.

Решения:

а) использование контейнерной архитектуры (Docker) для упрощения развертывания дополнительных экземпляров сервисов;

б) возможность горизонтального масштабирования серверной части и фоновых обработчиков задач;

в) оптимизация сетевого взаимодействия и конфигураций сервисов при увеличении нагрузки;

г) анализ и контроль использования вычислительных ресурсов.

Тестирование одновременной работы и риски

Решения:

2.1.2.3 Анализ требований по критериям отказов

Требования по критериям отказов определяют устойчивость системы к ошибкам, возникающим в процессе её эксплуатации, а также способность сохранять работоспособность при некорректных действиях пользователей, сбоях отдельных компонентов и ошибках обработки данных. Для системы информационной поддержки специалистов службы технической поддержки выполнение данных требований является обязательным, так как отказ системы может привести к невозможности оперативного получения необходимой информации.

Сохранение функциональности при пользовательских ошибках

Требование: ошибки, вызванные действиями пользователей (некорректный ввод данных, ошибочные запросы, попытки выполнения недопустимых операций), не должны приводить к отказу системы или её отдельных подсистем.

Данное требование распространяется на:

- ввод некорректных параметров при поиске информации;
- загрузку файлов неподдерживаемого формата;
- выполнение недопустимых операций пользователями с ограниченными правами доступа.

Решения:

а) реализовать валидацию входных данных на серверной стороне, включая проверку форматов, типов данных и допустимых значений;

б) использовать проверки прав доступа перед выполнением операций изменения данных;

в) формировать информативные сообщения об ошибках для пользователя без раскрытия внутренней логики системы;

г) предотвращать выполнение операций, нарушающих целостность данных.

Устойчивость программных интерфейсов (API)

Требование: программные интерфейсы системы должны оставаться работоспособными независимо от действий одного или нескольких пользователей. Ошибки отдельных запросов не должны нарушать функционирование API в целом.

Решения:

а) реализовать обработку исключений в серверной логике с использованием механизмов перехвата ошибок;

б) возвращать стандартизированные ответы об ошибках в формате JSON с указанием кода и описания ошибки;

в) изолировать обработку запросов разных пользователей, чтобы сбои в одной сессии не влияли на другие;

г) ограничивать частоту запросов при необходимости для предотвращения перегрузки системы.

Изоляция пользователей и сессий

Требование: ошибки, возникающие в рамках одной пользовательской сессии, не должны оказывать влияния на работу других пользователей и подсистем.

Решения:

а) использование механизмов сессионной изоляции на серверной стороне;

б) разделение контекста выполнения запросов пользователей;

в) обеспечение корректной обработки параллельных запросов без пересечения данных.

Обработка критических ошибок

Требование: при возникновении критических ошибок система должна сохранять работоспособность ключевых функций и обеспечивать возможность последующего анализа причин отказа.

Решения:

а) локализовать ошибки на уровне отдельных операций без остановки всей системы;

б) фиксировать все критические ошибки в журналах системы с указанием времени, типа ошибки и затронутого компонента;

в) обеспечивать корректное завершение некорректных операций без потери данных;

г) при необходимости выполнять повторную обработку операций в фоновом режиме.

Тестирование устойчивости к отказам

Для подтверждения соответствия системы требованиям по критериям отказов необходимо проводить тестирование устойчивости.

Решения:

а) моделирование пользовательских ошибок (ввод некорректных данных, прерывание операций);

б) тестирование работы системы при частичной недоступности компонентов;

в) проверка корректности работы API при ошибочных и неполных запросах;

г) анализ поведения системы после возникновения ошибок.

Риски и способы их минимизации

К основным рискам относятся:

- отказ отдельных функций при некорректных действиях пользователей;
- нарушение целостности данных при ошибках обработки запросов;
- деградация производительности при массовых ошибках пользователей.

Для минимизации указанных рисков необходимо:

а) применять строгую валидацию данных на всех этапах обработки;

б) изолировать выполнение пользовательских запросов;

в) использовать журналирование и мониторинг ошибок;

г) регулярно анализировать логи и устранять выявленные причины отказов.

2.1.3.1 Анализ требований к информационному обеспечению

Информационное обеспечение разрабатываемой системы определяет требования к составу, структуре, актуальности и сохранности данных, используемых в процессе функционирования системы. Точность и актуальность информации

Требование: информация, предоставляемая системой пользователям, должна быть точной, достоверной и актуальной. Все данные должны соответствовать фактическому состоянию и обновляться при изменении информации.

Ключевые аспекты:

а) недопустимо отображение устаревших или некорректных данных;

в) данные, используемые в системе, должны проходить проверку перед сохранением и отображением.

Решения:

а) реализация механизмов валидации данных при добавлении и редактировании информации;

б) централизованное хранение данных в базе данных PostgreSQL, обеспечивающее согласованность и целостность информации;

в) использование единой модели данных для всех компонентов системы.

Полнота представления информации

Требование: система должна обеспечивать полное и корректное представление информации без искажений независимо от используемого клиентского интерфейса.

Ключевые аспекты:

а) поддержка различных типов информации, включая текстовые материалы, изображения и файлы документации;

б) отсутствие потери данных или искажений при передаче и отображении информации;

в) единый формат представления информации для веб-интерфейса и Telegram-бота.

Решения:

а) хранение метаданных и файлов в централизованном хранилище с привязкой к записям базы данных;

б) контроль корректности загружаемых файлов и их форматов;

в) использование стандартных форматов данных при передаче информации.

Резервное копирование данных

Требование: информационное обеспечение системы должно предусматривать сохранность данных при сбоях и аварийных ситуациях.

Ключевые аспекты:

а) регулярность создания резервных копий данных;

б) полнота резервного копирования, включая пользовательские данные, документы и служебную информацию;

в) возможность восстановления данных в случае отказа.

Решения:

а) настройка регулярного резервного копирования базы данных PostgreSQL;

б) хранение резервных копий на отдельном носителе или сервере;

в) периодическая проверка корректности восстановления данных из резервных копий.

Централизованное хранение и управление данными

Требование: все информационные ресурсы системы должны храниться централизованно и управляться через единый интерфейс.

Ключевые аспекты:

а) исключение дублирования информации;

б) единые правила доступа к данным;

в) упрощение актуализации и сопровождения информации.

Решения:

а) использование централизованной базы данных для хранения всей структурированной информации;

б) реализация управления данными через административный веб-интерфейс;

в) разграничение прав доступа к информации в зависимости от роли пользователя.

Журналирование операций с данными

Требование: все операции, связанные с изменением и доступом к данным, должны фиксироваться для последующего анализа и аудита.

Ключевые аспекты:

а) фиксация действий пользователей при добавлении, изменении и удалении информации;

б) регистрация ошибок и нештатных ситуаций;

в) возможность анализа истории изменений данных.

Решения:

а) реализация журналирования действий пользователей на серверной стороне;

б) хранение логов в централизованном виде;

Защита информации

Требование: информационное обеспечение системы должно предусматривать защиту данных от утраты, искажения и несанкционированного доступа.

2.1.3.2 Анализ требований к программному обеспечению

2.1.3.2.1 Анализ требований к серверному программному обеспечению

Серверное программное обеспечение является ключевым компонентом разрабатываемой системы, так как именно оно обеспечивает обработку пользовательских запросов, управление данными, реализацию бизнес-логики и взаимодействие между всеми подсистемами. Анализ требований к серверному программному обеспечению выполнен на основе функциональных и нефункциональных требований, сформулированных в подразделе 1.4 настоящего отчёта.

Надёжность и устойчивость серверной части

Требование: серверное программное обеспечение должно обеспечивать устойчивую и корректную работу системы при штатных и повышенных нагрузках, а также при возникновении частичных отказов отдельных компонентов.

Ключевые аспекты:

а) серверная часть не должна зависеть от одного программного процесса или компонента;

б) сбои отдельных сервисов не должны приводить к полной остановке системы;

в) должна обеспечиваться возможность оперативного восстановления работоспособности.

Решения:

а) использование серверного фреймворка Django, обеспечивающего стабильную обработку HTTP-запросов и масштабируемость приложения;

б) разделение серверных компонентов на логические части (веб-сервер, фоновые задачи, кэш), что снижает влияние отказов отдельных компонентов;

в) использование Docker-контейнеров для изоляции и упрощения управления серверными сервисами.

Поддержка горизонтального масштабирования

Требование: серверное программное обеспечение должно поддерживать возможность масштабирования при увеличении числа пользователей и объёма обрабатываемых данных.

Ключевые аспекты:

а) отсутствие жёсткой привязки к одному серверу или экземпляру приложения;

б) возможность запуска нескольких экземпляров серверного приложения;

в) равномерное распределение нагрузки между компонентами системы.

Решения:

а) контейнеризация серверного приложения с возможностью развертывания нескольких экземпляров;

б) вынос хранения состояния и кэша за пределы серверного приложения (использование Redis);

в) масштабирование фоновых обработчиков задач Celery в зависимости от нагрузки.

Реализация бизнес-логики и обработка запросов

Требование: серверное программное обеспечение должно обеспечивать корректную реализацию бизнес-логики системы и обработку всех пользовательских запросов.

Ключевые аспекты:

а) обработка запросов веб-интерфейса и Telegram-бота должна выполняться централизованно;

б) бизнес-логика должна быть отделена от представления данных;

в) серверная часть должна обеспечивать контроль доступа к функциям системы.

Решения:

а) реализация бизнес-логики в серверной части с использованием архитектурных возможностей Django;

б) использование REST-подхода для взаимодействия клиентских компонентов с сервером;

в) реализация механизмов аутентификации и авторизации пользователей на серверной стороне.

Взаимодействие с базой данных и кэшем

Требование: серверное программное обеспечение должно обеспечивать надёжное и эффективное взаимодействие с системой хранения данных.

Ключевые аспекты:

а) обеспечение целостности данных при выполнении операций записи и обновления;

б) минимизация нагрузки на базу данных при частых запросах;

в) устойчивость к временным сбоям компонентов хранения данных.

Решения:

а) использование реляционной базы данных PostgreSQL для хранения основной информации;

б) применение транзакций для обеспечения целостности данных;

в) использование Redis в качестве кэша для хранения часто запрашиваемых данных и промежуточных состояний;

г) реализация механизмов повторного выполнения операций при временных ошибках доступа к данным.

Сопровождаемость и расширяемость серверного программного обеспечения

Требование: серверное программное обеспечение должно быть удобным в сопровождении и готовым к дальнейшему развитию.

Ключевые аспекты:

а) понятная структура кода и модульность;

б) возможность добавления нового функционала без переработки существующих компонентов;

в) наличие документации и комментариев в коде.

2.1.4.1 Анализ требований к показателям назначения

Время отклика на пользовательские действия

Требование: операции, выполняемые пользователем при взаимодействии с системой, такие как поиск информации, открытие документации, получение ответа через Telegram-бот и навигация по веб-интерфейсу, должны выполняться в течение 1–2 секунд при штатной нагрузке.

Данное требование обусловлено необходимостью оперативного получения информации специалистами службы технической поддержки, в том числе в условиях выездной работы.

Решения:

а) использовать кэширование часто запрашиваемых данных с применением Redis для снижения нагрузки на базу данных и ускорения обработки запросов;

б) оптимизировать запросы к базе данных PostgreSQL за счёт:

- использования индексов по ключевым полям;
- оптимизации операций выборки данных;
- минимизации количества сложных соединений таблиц;

в) сократить объём данных, передаваемых по сети, путём передачи только необходимой информации в формате JSON;

г) применять асинхронную обработку ресурсоёмких операций с использованием фоновых задач Celery;

д) при необходимости проводить нагрузочное тестирование системы для подтверждения соответствия установленным требованиям по времени отклика.

Доступность системы

Требование: доступность системы должна составлять не менее 99,5% в год, что соответствует максимально допустимому времени недоступности порядка 43 часов в год, включая плановые технические работы.

Высокая доступность системы необходима для обеспечения непрерывной информационной поддержки специалистов.

Решения:

а) использовать отказоустойчивую серверную архитектуру с возможностью перезапуска отдельных компонентов без остановки всей системы;

б) реализовать регулярное резервное копирование базы данных и пользовательских данных;

в) развернуть систему мониторинга состояния серверов и сервисов для оперативного выявления и устранения сбоев;

г) использовать контейнеризацию (Docker) для упрощения управления компонентами и повышения устойчивости системы.

Время восстановления после сбоев

Требование: время восстановления работоспособности системы должно составлять:

- до 15 минут при стандартных сбоях (перезагрузка сервисов, кратковременные сетевые проблемы);
- до 1 часа при серьёзных сбоях (отказ серверного оборудования, повреждение данных).

Решения:

а) автоматизировать процедуры запуска и восстановления компонентов системы с использованием контейнерной архитектуры;

б) предусмотреть наличие резервных копий данных и механизмов их оперативного восстановления;

в) разработать и документировать процедуры аварийного восстановления (Disaster Recovery Plan);

г) периодически проводить тестирование сценариев восстановления для выявления потенциальных узких мест.

Сохранение целевого назначения системы

Требование: система должна сохранять работоспособность и соответствовать своему целевому назначению на протяжении всего срока эксплуатации.

Решения:

а) обеспечивать регулярное обновление и сопровождение программного обеспечения;

б) поддерживать обратную совместимость функций при модернизации системы;

в) сохранять ключевой функционал, необходимый для пользователей, даже при изменении архитектуры или расширении системы.

Оптимизация использования ресурсов

Требование: система должна быть ресурсно-эффективной и обеспечивать стабильную работу при минимальных затратах вычислительных ресурсов.

Решения:

а) использовать контейнеризацию для эффективного распределения вычислительных ресурсов;

б) регулярно анализировать нагрузку на серверы и базу данных;

в) оптимизировать программный код и конфигурации сервисов;

г) предусматривать возможность масштабирования системы при увеличении нагрузки.

Поддержка модульного расширения

Требование: система должна быть готова к расширению функциональности и интеграции с внешними сервисами без существенной переработки существующих компонентов.

Решения:

а) использовать модульную архитектуру серверной части системы;

б) реализовать взаимодействие компонентов через REST API;

в) обеспечивать обратную совместимость интерфейсов при обновлениях;

г) предусматривать возможность подключения новых модулей без остановки системы.

2.1.4.2 Анализ требований к надежности

Требования к надёжности определяют способность системы сохранять работоспособность в течение заданного времени, а также корректно функционировать при возникновении сбоев, отказов отдельных компонентов и неблагоприятных внешних воздействий. Для системы информационной поддержки специалистов службы технической поддержки надёжность является критически важным показателем, так как отказ системы напрямую влияет на доступность служебной информации и эффективность работы пользователей.

Использование отказоустойчивой серверной архитектуры и резервного копирования

Требование: система должна быть построена с использованием отказоустойчивой серверной архитектуры и механизмов резервного копирования ключевых компонентов и данных.

Ключевые аспекты:

а) отказ одного из компонентов системы не должен приводить к полной остановке работы системы;

б) данные системы должны быть защищены от потери при программных и аппаратных сбоях;

в) должна обеспечиваться возможность восстановления работоспособности системы.

Решения:

а) использование контейнерной архитектуры (Docker), позволяющей изолировать компоненты системы и перезапускать их независимо друг от друга;

б) регулярное резервное копирование базы данных PostgreSQL с хранением резервных копий на отдельном носителе или сервере;

в) резервное копирование конфигурационных файлов и данных, используемых серверной частью системы;

г) периодическая проверка корректности восстановления данных из резервных копий.

Поддержка резервных каналов и устойчивости сетевого взаимодействия

Требование: система должна сохранять работоспособность при временных сбоях сетевого взаимодействия и нестабильных каналах связи.

Ключевые аспекты:

а) временные сетевые сбои не должны приводить к отказу всей системы;

б) система должна корректно обрабатывать обрывы соединений и повторные подключения;

в) взаимодействие с пользователями должно восстанавливаться автоматически после восстановления связи.

Решения:

а) реализация механизмов повторных попыток выполнения сетевых операций;

б) использование Redis для хранения промежуточных состояний и обеспечения устойчивости работы Telegram-бота;

в) обработка сетевых ошибок без остановки серверной части системы;

г) логирование сетевых сбоев для последующего анализа.

Использование защищённых и стабильных каналов связи

Требование: для обеспечения надёжности и безопасности взаимодействия все сетевые соединения должны быть защищены и устойчивы к сбоям.

Ключевые аспекты:

а) защита данных при передаче по сети;

б) минимизация риска потери данных при сбоях соединения;

в) устойчивость соединений к кратковременным разрывам.

Решения:

а) использование защищённых соединений HTTPS на основе TLS/SSL;

б) корректная обработка незавершённых запросов и повторная передача данных при необходимости;

в) контроль корректности сетевых соединений на серверной стороне.

Мониторинг состояния системы

Требование: система должна обеспечивать возможность контроля своего состояния и выявления потенциальных проблем на ранних этапах.

Ключевые аспекты:

а) необходимость мониторинга состояния серверов и сервисов;

б) контроль загрузки ресурсов и выявление узких мест;

в) оперативное обнаружение отказов и деградации производительности.

Решения:

а) ведение журналов работы серверных компонентов и ошибок;

б) анализ логов для выявления повторяющихся сбоев и нестабильных участков системы;

в) настройка уведомлений для административного персонала при возникновении критических ошибок.

Риски и меры по их снижению

К основным рискам, связанным с надёжностью системы, относятся:

- отказ серверного оборудования или программных компонентов;
- потеря данных вследствие сбоев или некорректных действий;
- нестабильность сетевого соединения.

Для минимизации указанных рисков необходимо:

а) регулярно выполнять резервное копирование и проверку его корректности;

б) обеспечивать изоляцию компонентов системы;

в) проводить анализ журналов и статистики отказов;

г) своевременно обновлять программное обеспечение и конфигурации.

2.1.4.3 Анализ требований по безопасности

Требования по безопасности определяют меры по защите программных, информационных и технических ресурсов системы от несанкционированного доступа, утечек данных, умышленных и случайных воздействий. Для системы информационной поддержки специалистов службы технической поддержки обеспечение безопасности является обязательным условием её эксплуатации, так как система содержит служебную информацию и документацию ограниченного доступа.

Физическая безопасность серверного оборудования

Требование: серверное оборудование и компоненты системы должны размещаться в защищённых помещениях с ограниченным доступом.

Анализ: физический доступ к серверному оборудованию может привести к утечке данных, повреждению или отключению системы, поэтому обеспечение физической безопасности является базовым уровнем защиты.

Решения:

а) размещение серверов в помещениях с ограниченным доступом;

б) ограничение перечня лиц, имеющих физический доступ к оборудованию;

в) ведение учёта и контроля доступа к серверным помещениям.

Обеспечение безопасных условий эксплуатации оборудования

Требование: оборудование должно эксплуатироваться в условиях, соответствующих техническим требованиям производителей.

Анализ: несоблюдение температурного режима и условий влажности может привести к сбоям в работе оборудования и потере данных.

Решения:

а) обеспечение надлежащей вентиляции и охлаждения серверного оборудования;

б) контроль температурных режимов и состояния оборудования;

в) своевременное техническое обслуживание серверных компонентов.

Защита данных при передаче

Требование: все данные, передаваемые между компонентами системы и пользователями, должны быть защищены от перехвата и подмены.

Анализ: передача данных по незащищённым каналам связи создаёт риск атак типа «человек посередине» и утечки конфиденциальной информации.

Решения:

а) использование защищённых соединений HTTPS с применением протоколов TLS/SSL;

б) регулярное обновление сертификатов безопасности;

в) запрет использования незащищённых соединений для доступа к системе.

Использование защищённого удалённого доступа

Требование: удалённый доступ к серверной части системы должен осуществляться по защищённым каналам связи.

Анализ: удалённый доступ без защиты значительно увеличивает риск несанкционированного проникновения и компрометации системы.

Решения:

а) использование защищённых каналов удалённого доступа (VPN);

б) ограничение доступа к серверной инфраструктуре только для авторизованных пользователей;

в) применение многофакторной аутентификации для административного доступа.

Разграничение прав доступа пользователей

Требование: система должна обеспечивать разграничение прав доступа пользователей в зависимости от их ролей и функциональных обязанностей.

Анализ: неправильное распределение прав доступа может привести к несанкционированному изменению или удалению данных.

Решения:

а) реализация ролевой модели доступа (RBAC);

б) ограничение доступа пользователей только к тем функциям и данным, которые необходимы для выполнения их задач;

в) контроль и аудит изменений прав доступа.

Контроль действий пользователей и аудит безопасности

Требование: все действия пользователей, связанные с доступом и изменением данных, должны фиксироваться для последующего анализа.

Анализ: журналирование действий пользователей позволяет выявлять попытки несанкционированного доступа и анализировать инциденты безопасности.

Решения:

а) ведение журналов авторизации и действий пользователей;

б) хранение логов в централизованном виде;

в) регулярный анализ журналов безопасности административным персоналом.

Риски и меры по их снижению

К основным рискам в области безопасности относятся:

- несанкционированный доступ к данным;
- утечка служебной информации;
- компрометация учетных записей пользователей;
- физическое повреждение оборудования.

Для снижения указанных рисков необходимо:

а) использовать защищённые каналы передачи данных и VPN;

б) применять многофакторную аутентификацию;

в) регулярно обновлять программное обеспечение и сертификаты безопасности;

г) проводить аудит безопасности и анализ журналов.

2.1.4.4 Анализ требований к эргономике и технической эстетике

Требования к эргономике и технической эстетике определяют удобство использования системы, наглядность представления информации и снижение когнитивной нагрузки на пользователей. Для системы информационной поддержки специалистов службы технической поддержки данные требования имеют важное значение, так как система используется в условиях ограниченного времени и повышенной ответственности за принимаемые решения.

Доступность и логичность элементов управления

Требование: элементы управления системы должны быть расположены логично и обеспечивать быстрый доступ к основным функциям без необходимости выполнения большого количества действий.

Анализ: сложная или перегруженная навигация может приводить к ошибкам пользователей и увеличению времени выполнения операций, что негативно влияет на эффективность работы специалистов.

Решения:

а) проектирование интерфейса с интуитивно понятной навигацией и логической группировкой функций;

б) применение принципа «минимального количества действий» для выполнения типовых операций;

в) выделение основных функций на главных экранах системы.

Использование визуальных подсказок и структуры интерфейса

Требование: интерфейс системы должен обеспечивать наглядное представление информации с использованием визуальных подсказок, иерархии элементов и цветовой маркировки.

Анализ: использование визуальных ориентиров снижает вероятность ошибок и ускоряет восприятие информации пользователем.

Решения:

а) применение цветовой индикации для выделения важных элементов и сообщений;

б) использование иконок и графических обозначений для основных действий;

в) структурирование информации с использованием заголовков, списков и разделителей.

Минимализм и отсутствие избыточных элементов

Требование: интерфейс системы не должен быть перегружен второстепенной информацией и элементами, не влияющими на выполнение основных задач.

Анализ: избыточное количество элементов повышает когнитивную нагрузку и затрудняет работу с системой, особенно в стрессовых ситуациях.

Решения:

а) исключение второстепенных элементов с основных экранов;

б) использование лаконичного дизайна без лишних декоративных элементов;

в) предоставление дополнительной информации по запросу пользователя.

Адаптивность пользовательского интерфейса

Требование: система должна корректно отображаться и функционировать на устройствах с различными размерами экранов.

Анализ: пользователи системы могут использовать различные устройства, включая настольные компьютеры и мобильные устройства, что требует адаптивного интерфейса.

Решения:

а) применение адаптивного дизайна для веб-интерфейса;

б) тестирование интерфейса на устройствах с различными разрешениями экранов;

в) обеспечение корректной работы интерфейса при изменении размеров окна браузера.

Читаемость и визуальная согласованность

Требование: интерфейс системы должен обеспечивать высокую читаемость текста и визуальную согласованность элементов.

Анализ: низкая читаемость текста и несогласованность элементов затрудняют восприятие информации и увеличивают вероятность ошибок.

Решения:

а) использование читаемых шрифтов и достаточного размера текста;

б) применение единой цветовой схемы и стиля элементов интерфейса;

в) обеспечение достаточного контраста между текстом и фоном.

Предотвращение пользовательских ошибок

Требование: интерфейс должен минимизировать вероятность совершения пользователями ошибочных действий.

Анализ: предупреждение ошибок на уровне интерфейса снижает нагрузку на систему и повышает надёжность работы пользователей.

Решения:

а) использование подтверждающих диалогов перед выполнением критических операций;

б) отображение подсказок и сообщений о возможных ошибках;

в) информирование пользователя о результатах выполненных действий.

Соответствие эргономическим стандартам

Требование: пользовательский интерфейс системы должен соответствовать общепринятым требованиям эргономики и удобства использования.

Анализ: соблюдение эргономических принципов повышает качество пользовательского опыта и снижает утомляемость пользователей.

Решения:

а) использование общепринятых шаблонов и стандартов проектирования интерфейсов;

б) проведение анализа удобства использования интерфейса;

в) учёт рекомендаций по доступности и эргономике при проектировании интерфейса.

К основным рискам в области эргономики и технической эстетики относятся:

- сложность освоения системы новыми пользователями;
- увеличение количества пользовательских ошибок;
- снижение эффективности работы из-за неудобного интерфейса.

Для снижения указанных рисков необходимо:

а) регулярно анализировать отзывы пользователей;

б) вносить улучшения в интерфейс на основе выявленных проблем;

в) тестировать удобство использования системы на практике.

2.1.4.5 Анализ требований к защите информации от несанкционированного доступа

Требования к защите информации от несанкционированного доступа направлены на предотвращение получения, изменения или уничтожения данных лицами, не имеющими соответствующих прав. Для системы информационной поддержки специалистов службы технической поддержки защита информации является обязательным условием эксплуатации, поскольку система содержит служебные материалы, инструкции и иную информацию ограниченного доступа.

Использование уникальных учетных записей пользователей

Требование: каждый пользователь системы должен иметь уникальную учетную запись для доступа к функционалу системы.

Анализ: использование уникальных учетных записей обеспечивает идентификацию пользователей и позволяет отслеживать их действия в системе.

Решения:

а) реализация механизма аутентификации пользователей на основе логина и пароля;

б) запрет использования общих учетных записей;

в) хранение учетных данных в зашифрованном виде с использованием современных криптографических алгоритмов (PBKDF2, Argon2).

Политика паролей и аутентификации

Требование: система должна обеспечивать надёжную аутентификацию пользователей и защиту учетных данных.

Анализ: слабые пароли и отсутствие дополнительных механизмов защиты увеличивают риск компрометации учетных записей.

Решения:

а) установка требований к сложности паролей (минимальная длина, использование различных типов символов);

б) реализация механизма регулярной смены паролей;

в) ограничение количества неудачных попыток входа в систему.

Многофакторная аутентификация для пользователей с расширенными правами

Требование: для пользователей с административными правами должен использоваться дополнительный фактор аутентификации.

Анализ: использование многофакторной аутентификации значительно снижает риск несанкционированного доступа при компрометации пароля.

Решения:

а) применение многофакторной аутентификации для административных учетных записей;

б) использование одноразовых кодов или внешних средств подтверждения доступа;

в) реализация резервных методов восстановления доступа.

Разграничение прав доступа на основе ролей

Требование: доступ пользователей к функциям и данным системы должен быть ограничен в соответствии с их ролями и функциональными обязанностями.

Анализ: отсутствие чёткого разграничения прав доступа может привести к несанкционированному изменению или удалению данных.

Решения:

а) реализация ролевой модели доступа (RBAC);

б) предоставление пользователям минимально необходимых прав;

в) контроль и аудит изменений прав доступа.

Защита конфиденциальных данных при хранении

Требование: конфиденциальные данные системы должны быть защищены от несанкционированного доступа при хранении.

Анализ: при компрометации серверного оборудования или носителей данных возможна утечка информации.

Решения:

а) использование шифрования данных при хранении, включая учетные данные пользователей;

б) хранение ключей шифрования отдельно от основных данных;

в) ограничение доступа к данным на уровне серверной инфраструктуры.

Логирование и аудит действий пользователей

Требование: все действия пользователей, связанные с доступом к системе и изменением данных, должны фиксироваться.

Анализ: журналирование действий пользователей позволяет выявлять попытки несанкционированного доступа и анализировать инциденты безопасности.

Решения:

а) ведение журналов авторизации, выхода и изменения данных;

б) хранение логов в централизованном виде с ограниченным доступом;

в) регулярный анализ журналов безопасности административным персоналом.

К основным рискам относятся:

- компрометация учетных записей пользователей;
- несанкционированный доступ к данным;
- утечка конфиденциальной информации;
- злоупотребление правами доступа.

Для снижения указанных рисков необходимо:

а) использовать сложные пароли и многофакторную аутентификацию;

б) строго разграничивать права доступа пользователей;

в) регулярно анализировать журналы безопасности;

г) своевременно обновлять программное обеспечение и средства защиты.

2.1.4.6 Анализ требований по сохранности информации при авариях

Требования по сохранности информации при авариях направлены на обеспечение защиты данных от утраты и искажения в случае возникновения нештатных ситуаций, таких как программные сбои, отказ оборудования, ошибки персонала или внешние воздействия. Для системы информационной поддержки специалистов службы технической поддержки выполнение данных требований является критически важным, поскольку утрата служебной информации может привести к остановке рабочих процессов и снижению эффективности работы пользователей.

Автоматизированное создание резервных копий данных

Требование: система должна обеспечивать регулярное и автоматизированное создание резервных копий данных.

Анализ: ручное резервное копирование не гарантирует своевременность и полноту сохранения данных, особенно при интенсивной эксплуатации системы.

Решения:

а) организация автоматического резервного копирования базы данных PostgreSQL по заданному расписанию;

б) использование инкрементальных резервных копий для ежедневного сохранения изменений и полных резервных копий на регулярной основе;

в) хранение резервных копий на отдельном сервере или носителе, физически или логически изолированном от основной системы;

г) периодическая проверка корректности создаваемых резервных копий.

Ведение журналов аварийных ситуаций и восстановления

Требование: все аварийные ситуации и действия по восстановлению системы должны фиксироваться в журналах.

Анализ: наличие журналов аварий позволяет анализировать причины сбоев и оценивать эффективность принятых мер по восстановлению.

Решения:

а) ведение журналов событий, связанных с отказами компонентов системы;

б) фиксация времени возникновения аварии, затронутых компонентов и предпринятых действий;

в) хранение журналов в централизованном виде для последующего анализа.

Защита резервных копий данных

Требование: резервные копии данных должны быть защищены от несанкционированного доступа и утраты.

Анализ: резервные копии содержат полный объём информации системы и при утечке представляют серьёзную угрозу безопасности.

Решения:

а) использование шифрования резервных копий данных;

б) ограничение доступа к резервным копиям только для авторизованного административного персонала;

в) защита каналов передачи резервных копий при их создании и хранении.

Приоритетное восстановление критически важных функций

Требование: в случае аварии восстановление системы должно выполняться с учётом приоритетности функций.

Анализ: не все функции системы имеют одинаковую важность для обеспечения её работоспособности. Ключевые функции должны восстанавливаться в первую очередь.

Решения:

а) определение перечня критически важных функций системы (доступ к документации, поиск информации, работа Telegram-бота);

б) разработка последовательности восстановления компонентов системы;

в) обеспечение возможности быстрого восстановления базовой функциональности до полного восстановления системы.

Процедуры аварийного восстановления

Требование: для системы должны быть разработаны и задокументированы процедуры аварийного восстановления.

Анализ: отсутствие формализованных процедур увеличивает время восстановления системы и повышает риск потери данных.

Решения:

а) разработка плана аварийного восстановления (Disaster Recovery Plan);

б) описание последовательности действий при различных типах аварий;

в) периодическое тестирование процедур восстановления в контролируемых условиях;

г) актуализация процедур при изменении архитектуры системы.

К основным рискам, связанным с сохранностью информации при авариях, относятся:

- потеря данных вследствие отказа оборудования или программных сбоев;
- повреждение резервных копий;
- увеличение времени восстановления системы.

Для минимизации указанных рисков необходимо:

а) регулярно выполнять и проверять резервное копирование данных;

б) использовать избыточность хранения резервных копий;

в) шифровать резервные данные и ограничивать к ним доступ;

г) проводить тестирование процедур аварийного восстановления.
